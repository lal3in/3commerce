# OWASP ASVS L1 Self-Audit — 3commerce

Pre-launch security self-audit of the custom auth (Identity) and the gateway, against
OWASP ASVS Level 1. Scope: ADR-0012 conditions, PRD §9. Status legend: ✅ pass · ⚠️ partial/follow-up · ⬜ deferred (post-registration) · n/a.

> This is a **self-audit**, not a substitute for the external review that is a launch gate
> (PRD Appendix B). It records the current posture so findings are tracked, not lost.

## V2 — Authentication

| # | Requirement | Status | Evidence |
|---|-------------|--------|----------|
| 2.1.1 | Passwords ≥ 8 (we require ≥ 10) | ✅ | `RegisterRequest`/`ResetConfirmRequest` `MinLength(10)` |
| 2.1.7 | Passwords checked against breach lists | ⬜ | deferred (HIBP integration post-MVP) |
| 2.2.1 | Anti-automation / rate limiting on auth | ✅ | gateway per-IP limiter on `/login`,`/register`,`/password-reset` (30/min) + per-account progressive lockout |
| 2.2.3 | Secure notifications on security events | ⚠️ | verification/reset emails sent; no "new login" alerts (v1) |
| 2.4.1 | Passwords stored with approved KDF | ✅ | Argon2id m=19456,t=2,p=1 (OWASP first-choice) via vetted library; never hand-rolled |
| 2.4.4 | Per-credential salt | ✅ | 16-byte CSPRNG salt per hash, self-describing encoding |
| 2.5.1 | No default/shared credentials | ✅ | dev admin seeded only in Development from config; no prod defaults |
| 2.5.4 | Account recovery doesn't reveal account existence | ✅ | reset request returns identical response regardless of existence |
| 2.7.x | Out-of-band verifier (email tokens) single-use, expiring | ✅ | `EmailToken` single-use, 24h (verify) / 1h (reset), stored hashed |

## V3 — Session Management

| # | Requirement | Status | Evidence |
|---|-------------|--------|----------|
| 3.2.1 | Sessions generated by a trusted RNG | ✅ | 256-bit `RandomNumberGenerator` opaque token |
| 3.2.3 | Token stored hashed at rest | ✅ | only SHA-256 of the token in `Sessions.TokenHash` |
| 3.3.1 | Logout invalidates the session | ✅ | revoke row; gateway cache ≤ 60s bound (NFR-8) |
| 3.3.2 | Re-auth / revocation on credential change | ✅ | password reset revokes all sessions |
| 3.4.1 | Cookies `Secure` (when HTTPS) | ✅ | `Secure = IsHttps`, `HttpOnly`, `SameSite=Lax` |
| 3.4.2 | Cookies `HttpOnly` | ✅ | always set |
| 3.5.2 | No static API secrets in the browser | ✅ | opaque cookie only; no JWT/keys client-side |

## Open follow-ups (before live traffic)

1. ⬜ External penetration test / review (launch gate, PRD Appendix B).
2. ⬜ Breached-password check (HIBP) at registration/reset.
3. ⚠️ Rotate all dev secrets; move to a secret manager for any deployed environment.
4. ⬜ MFA for admin accounts (PRD §13).
5. ⚠️ "New device/login" security notifications.

No **critical** open findings in the current pre-launch scope. Re-run this audit after any auth change.
