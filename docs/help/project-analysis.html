3commerce

1.What 3commerce is

3commerce is a ground-up e-commerce platform with two stated, equal goals: be a serious learning exercise in distributed-systems engineering, and be a real, launchable storefront a solo operator could put live. That dual mandate (codified in ADR-0001) explains nearly every decision on the page below — it is why the system reaches for production-grade patterns (sagas, outbox, double-entry ledger, custom auth) that a pure MVP would skip, and also why some of those patterns are wired but not yet load-bearing.

The shape of the system

  • Backend: 6 C# services — Identity, Catalog, Ordering, Payments, Fulfillment, Support — each owning its own PostgreSQL database.
  • Messaging: async via MassTransit + RabbitMQ, with an EF transactional outbox and saga state machines (checkout, RMA).
  • Edge: a YARP gateway as the single public origin; an opaque 3c_session cookie is exchanged for an internal ES256 claims JWT minted gateway-side.
  • Money: an append-only double-entry ledger with DB-enforced balance and append-only triggers; IPaymentProvider (real Stripe + FakePaymentProvider); a single RefundRequested refund path.
  • Frontends: a Next.js 15 SSR storefront (:3000) and a Blazor Server admin console (:5200), both thin and gateway-only.
  • Assurance: 11 unit + 25 integration (Testcontainers) + 13 Playwright E2E tests, an e2e-verify.sh regression script, and a green multi-job CI matrix.
  • Docs: a 16-section PRD, 20 ADRs, 6 OpenAPI specs, an ASVS L1 audit, an MVP runbook, and the operator wiki this page belongs to.

Operator-facing docs for these components live in the help wiki (getting-started, storefront, admin, testing, deployment).

3.Architecture analysis

Microservices — chosen knowingly against the safer default

The most consequential decision is also the most debatable: 3commerce is 6 microservices from day one (ADR-0005/ADR-0006), even though the conventional advice for a solo builder is "modular monolith first, extract services later." This was a deliberate choice driven by the learning goal, and it should be judged as such rather than as an accident.

What it buys

Real distributed-systems practice: capability seams, independent schemas, async contracts, saga orchestration, and the operational reality of a message bus — exactly the things you cannot learn from a monolith.

What it costs

Six deployables, six DBs, a bus, and a gateway to run for what an MVP could ship as one process. The cost shows up downstream: the app tier still isn't containerized, k8s is deferred, and the operational surface is larger than the feature set strictly needs.

The honest verdict: for a learning-first project the split is defensible and well executed (clean DB-per-service isolation, event-fed read models, zero out-of-scope creep). For a business-first MVP it is over-engineered — but the project never pretended otherwise.

Event-driven core: outbox + sagas

Async messaging is done by the book: an EF transactional outbox means state changes and the events announcing them commit atomically, and saga state machines coordinate multi-step flows (checkout, RMA) without distributed transactions. The dedup inbox makes consumers idempotent. This is the right pattern and it is implemented faithfully.

Caveat — wired, lightly proven

The saga scheduler is in-memory Quartz, and the checkout saga's resilience is largely untested: the only chaos test runs against the ping-pong spine (NFR-2 Partial). A code comment in CheckoutStateMachine claims out-of-order webhook tolerance, but the code actually requires CartSubmitted first — a candid honesty gap admitted as future hardening.

Gateway + auth model

A YARP gateway is the single public origin (ADR-0011). Browsers hold only an opaque 3c_session cookie; the gateway exchanges it for a short-lived ES256 internal-claims JWT, with the private key held gateway-side only and inbound claim headers stripped. This is a textbook BFF/edge-auth design and one of the project's strongest pieces (see §5).

Ledger as source of truth

Money does not live in mutable order rows; it lives in an append-only double-entry ledger whose invariants are enforced in the database — a deferred balance-constraint trigger, append-only triggers, and line-level check constraints. The application cannot write an unbalanced or back-dated entry even if it tries. Trial-balance-equals-zero is verified in tests and in live E2E. This is the architectural decision that most elevates the project.

The two-language frontend (Next.js + Blazor) is the most questionable stack choice for a solo team. 5.The best of this project

★ DB-enforced double-entry ledger

Integrity is enforced where it can't be bypassed — at the database. A deferred balance constraint, append-only triggers, and line check-constraints make unbalanced or mutated entries impossible, and trial-balance-=-0 is verified in both tests and live E2E. This is production-grade financial discipline most startups never reach.

★ A single, honest refund path

Every refund — customer RMA or operator action — funnels through one idempotent RefundRequested / ExecuteRefundConsumer, with an over-refund guard, proportional tax reversal, and a double-approve no-op. No side doors, no ad-hoc money movement.

★ Seam discipline — fakes that reuse the real path

/dev/simulate-payment flows through the same PaymentEventProcessor and dedup inbox as the real Stripe webhook (and is gated to Development). The dev fakes don't bypass the system — they exercise it, so what you test in dev is what runs in prod.

★ Custom auth, done correctly

Argon2id (m=19456, t=2, p=1), opaque cookie → ES256 claims with a gateway-only private key, inbound header-strip, and ≤60s revocation — all test-backed. A genuinely hard thing to get right, and it is right.

★ Test depth & green CI

11 unit + 25 integration (against real infra) + 13 Playwright E2E across both frontends, an e2e-verify.sh L1–L20 regression script, and a green build/integration/browser/docker CI matrix. Real-browser admin E2E in particular is rare for a project this size.

★ Documentation rigor

16-section PRD, 20 ADRs, 6 OpenAPI specs, an ASVS L1 security audit, and a runbook. The reasoning behind every major decision is written down — which is precisely what makes an honest conformance review possible in the first place.

6.The weakest parts

FR-7 missing — and the status file says it's done

Post-purchase guest→account conversion simply doesn't exist: no service method, no endpoint, no page, and guest orders never attach to a later account. Worse than the gap is the honesty mismatch — a status file overstates it as complete. This is the single most important thing to fix, in code and in the tracking docs.

Placeholder admin & account screens

The admin orders screen is a placeholder (explanatory text, no list/detail), there's no catalog CRUD (FR-12 Partial), and the customer account page has no order history (just email + verified flag). The storefront is well ahead of the back office.

UI honesty gap — build-validated only

Several UI flows are validated by compiling/building, not by exercising behavior. Combined with the CheckoutStateMachine comment that overstates webhook tolerance, the project's claims occasionally run slightly ahead of its proof.

Untested NFRs (2 / 5 / 7)

Saga chaos resilience, product-SSR p95, and the single end-to-end checkout trace are all wired but unmeasured by tests. (Search p95 is measured — proof that the harness can do it.) The targets exist; the evidence doesn't yet.

Free-form RMA refund amount

The RMA flow takes the refund amount as free-form customer input ("Amount in cents"). The over-refund guard catches the worst case, but accepting customer-supplied money figures is a trust/validation smell that ought to derive the amount server-side.

No app-tier containerization · dev secrets committed · no k8s

Only the 6 backend services + gateway + worker have Dockerfiles; the storefront and admin run via bare dotnet run/npm (no Dockerfiles). A dev ES256 private key and admin password are committed (clearly labelled DEV-ONLY) and STORE_CURRENCY is hard-coded to "EUR" rather than wired despite ADR-0015 referencing it. k8s and production are deferred. These are launch gates, not engine flaws — but they're real.

Minor drift to clean up

No nav links to /orders/[id]/support (the page works, nothing points to it); the admin console verifies the admin role by side-effect (probing a Xero endpoint) rather than a dedicated check; and docs lag reality (AGENTS.md says "L1–L19 / 11 unit + 25 integration" while the script is now L1–L20). Small, but they erode trust in the written record.

10.Verdict

3commerce is an exceptional learning artifact and a genuinely solid engineering core, wrapped in an MVP that isn't finished. The A− is well earned and well placed.

What it is excellent at: the money engine. A DB-enforced double-entry ledger, a single idempotent refund path, seam discipline where dev fakes exercise the real path, by-the-book custom auth, high-fidelity tests with real browsers and green CI, and documentation rigor most teams never approach. If you want to learn event-driven .NET — outbox, sagas, gateway auth, ledgers — this is a near-ideal reference.

What it is not (yet): a turnkey, launchable business. FR-7 is missing (and mislabelled as done), the admin and account screens are partly placeholders, key NFRs are wired but unmeasured, the RMA amount is customer-supplied, the app tier isn't containerized, dev secrets are committed, and k8s/production are deferred. These are launch gates and honesty cleanups — bounded, well-understood, and none of them touch the ledger.

Who it's a good template for: engineers and teams who want a rigorous, well-documented blueprint for distributed e-commerce in modern .NET — provided they treat the frontend/back-office completeness and the launch plumbing as the remaining work, and tighten the few places where the written record runs ahead of the code.
