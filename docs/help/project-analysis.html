3commerce — Project analysis & honest assessment

A from-scratch, distributed e-commerce platform built deliberately as both a learning vehicle and a launchable business — six .NET 10 microservices, an event-driven money core, and a two-language frontend estate. This is a full re-audit: every requirement re-walked against the current code after the post-MVP backlog (BL-1…BL-11) and the new containerized launch track landed.

What 3commerce is

3commerce is a ground-up e-commerce platform with two stated, equal goals: be a serious learning exercise in distributed-systems engineering, and be a real, launchable storefront a solo operator could put live. That dual mandate (codified in ADR-0001) explains nearly every decision on the page below — it is why the system reaches for production-grade patterns (sagas, outbox, double-entry ledger, custom auth) that a pure MVP would skip.

The shape of the system

  • Backend: 6 C# services — Identity, Catalog, Ordering, Payments, Fulfillment, Support — each owning its own PostgreSQL database.
  • Messaging: async via MassTransit + RabbitMQ, with an EF transactional outbox and saga state machines (checkout, RMA).
  • Edge: a YARP gateway as the single public origin; an opaque 3c_session cookie is exchanged for an internal ES256 claims JWT minted gateway-side.
  • Money: an append-only double-entry ledger with DB-enforced balance and append-only triggers; IPaymentProvider (real Stripe + FakePaymentProvider); a single server-derived RefundRequested refund path.
  • Frontends: a Next.js 15 SSR storefront (:3000) and a Blazor Server admin console (:5200), both thin and gateway-only.
  • Launch: a dual model — bare dotnet run for the inner loop, plus a full docker-compose.yml stack and a Helm umbrella chart, wrapped by scripts/launch.sh (--fresh/--reuse × --env dev/prod).
  • Assurance: ~14 unit + ~36 integration (Testcontainers) + 13 Playwright E2E tests, an e2e-verify.sh regression script, and a green multi-job CI matrix (incl. compose-smoke + kind-deploy).
  • Docs: a 16-section PRD, 21 ADRs, OpenAPI specs, an ASVS L1 audit, an MVP runbook, and the operator wiki this page belongs to.

Operator-facing docs for these components live in the help wiki (getting-started, storefront, admin, testing, deployment).

New: a repeatable containerized launch

The most significant change since the last audit is operational. ADR-0021 adds a dual launch model on top of the bare-run inner loop, so the whole estate can be brought up the way it would actually deploy.

+

★ Compose → Helm/k8s, fresh or reuse, dev or prod — A hardened docker-compose.yml runs all 10 app images + infra on one network (healthchecks, depends_on, resource limits, env-files), wrapped by scripts/launch.sh [--fresh|--reuse] [--env dev|prod]. The same topology is packaged as a Helm umbrella chart (deploy/helm/3commerce) and validated against a kind cluster in CI. Migrations run as a one-shot migrate image of framework-dependent EF bundles (deploy/migrator) with design-time DbContext factories — decoupled from app boot and secrets. --fresh drops the pgdata volume so init-SQL and migrations re-run; --env prod mints a fresh ES256 keypair and enforces the BL-11 secret gate.

This closes the loop the prior audit flagged as "app tier not containerized / k8s deferred": both frontends now have Dockerfiles, the full stack composes, and a chart deploys. What remains deferred is genuinely external (below), not plumbing.

Architecture analysis

Microservices — chosen knowingly against the safer default

The most consequential decision is also the most debatable: 3commerce is 6 microservices from day one (ADR-0005/ADR-0006), even though the conventional advice for a solo builder is "modular monolith first, extract services later." This was a deliberate choice driven by the learning goal, and it should be judged as such.

+

What it buys — Real distributed-systems practice: capability seams, independent schemas, async contracts, saga orchestration, and the operational reality of a message bus — exactly the things you cannot learn from a monolith.

!

What it costs — Ten deployables, six DBs, a bus, and a gateway to run for what an MVP could ship as one process. The operational surface is larger than the feature set strictly needs — though it is now fully containerized and chart-deployable, so the cost is paid in run-cost and complexity rather than in missing plumbing.

The honest verdict: for a learning-first project the split is defensible and well executed (clean DB-per-service isolation, event-fed read models, zero out-of-scope creep). For a business-first MVP it is over-engineered — but the project never pretended otherwise.

Event-driven core: outbox + sagas

Async messaging is done by the book: an EF transactional outbox means state changes and the events announcing them commit atomically, and saga state machines coordinate multi-step flows (checkout, RMA) without distributed transactions. The dedup inbox makes consumers idempotent. The resilience claim is no longer just mechanism-deep: a chaos test restarts the saga host mid-checkout and the saga still reaches a correct, balanced terminal state.

!

Caveat — durability of the scheduler — The saga timeout scheduler is in-memory Quartz, so scheduled timeouts don't survive a process restart (the chaos test exercises the durable-queue path, not a pending timeout across a restart). A production-durable scheduler is a known future hardening, not a correctness gap on the happy/declined paths.

Gateway + auth model

A YARP gateway is the single public origin (ADR-0011). Browsers hold only an opaque 3c_session cookie; the gateway exchanges it for a short-lived ES256 internal-claims JWT, with the private key held gateway-side only and inbound claim headers stripped. This is a textbook BFF/edge-auth design and one of the project's strongest pieces (see The best of this project). A new dev-secret gate (DevSecretGuard) refuses to boot outside Development with the committed dev key.

Ledger as source of truth

Money does not live in mutable order rows; it lives in an append-only double-entry ledger whose invariants are enforced in the database — a deferred balance-constraint trigger, append-only triggers, and line-level check constraints. The application cannot write an unbalanced or back-dated entry even if it tries. Trial-balance-equals-zero is verified in tests and in live E2E. This is the architectural decision that most elevates the project.

The best of this project

+

★ DB-enforced double-entry ledger — Integrity is enforced where it can't be bypassed — at the database. A deferred balance constraint, append-only triggers, and line check-constraints make unbalanced or mutated entries impossible, and trial-balance-=-0 is verified in both tests and live E2E. This is production-grade financial discipline most startups never reach.

+

★ A single, server-derived refund path — Every refund — customer RMA or operator action — funnels through one idempotent RefundRequested / ExecuteRefundConsumer. The RMA amount is now derived server-side from an OrderSnapshot read-copy (summed per selected line, capped at purchased quantity); the client never supplies a trusted figure. Over-refund guard, proportional tax reversal, and a double-approve no-op round it out.

+

★ Seam discipline — fakes that reuse the real path — /dev/simulate-payment flows through the same PaymentEventProcessor and dedup inbox as the real Stripe webhook (and is gated to Development). The dev fakes don't bypass the system — they exercise it, so what you test in dev is what runs in prod.

+

★ Custom auth, done correctly — Argon2id (m=19456, t=2, p=1), opaque cookie → ES256 claims with a gateway-only private key, inbound header-strip, ≤60s revocation, and a fail-fast DevSecretGuard against shipping the committed dev key — all test-backed. A genuinely hard thing to get right, and it is right.

+

★ Test depth & green CI — ~14 unit + ~36 integration (against real Postgres/RabbitMQ) + 13 Playwright E2E across both frontends, an e2e-verify.sh regression script, and a green build/integration/browser/docker + compose-smoke + kind-deploy CI matrix. The new chaos and single-trace tests close the prior "wired but unmeasured" gaps.

+

★ Documentation rigor — 16-section PRD, 21 ADRs, OpenAPI specs, an ASVS L1 security audit, a runbook, and a secrets/rotation doc. The reasoning behind every major decision is written down — which is precisely what makes an honest conformance re-audit possible in the first place.

What's still open (launch gates, not engine flaws)

With every code gap from the prior audit closed, the remaining work is deliberately external or a measurement nuance — none of it sits in the money or auth core.

!

External integrations await a legal entity — Stripe and Xero run against an interface-seamed FakePaymentProvider / LoggingXeroClient, and ITaxStrategy is a flat-rate placeholder. Going live needs company registration → real Stripe/Xero credentials and a real tax strategy. The seams (IPaymentProvider, IXeroClient, ITaxStrategy) make these swaps, not rewrites.

!

NFR-5 SSR p95 is proxied, not isolated — The SSR-dominant product-detail API fetch is benchmarked under a 500ms p95 budget, but the literal rendered-page p95 (< 800ms target) isn't measured at the Next.js render boundary. Close, and almost certainly satisfied — but the exact metric is a proxy.

!

Prod DB-credential rotation deferred — Prod launch rotates app secrets (ES256 keypair + admin password via the BL-11 gate), but local Postgres/RabbitMQ creds in compose/k8s remain the init-SQL dev passwords. Real cluster-credential rotation and a secrets store are explicitly deferred (ADR-0021).

!

kind-deploy CI is slow and cost-gated; external pen test pending — The kind cluster deploy is the slowest, flakiest CI job (paths-gated, helm --wait). And an external penetration test remains a pre-launch gate — the ASVS L1 self-audit is internal.

i

Minor housekeeping — The status file still marks cl_8 (launch.sh) and a few doc tasks pending even though the script and artifacts exist on disk — stale markers, verified by inspection, not real gaps. The saga timeout scheduler remains in-memory Quartz (future-durable).

Verdict

Verdict

3commerce is an exceptional learning artifact and a genuinely solid engineering core — and, after the post-MVP backlog and the containerized launch track, it is now a near-complete MVP. The grade rises to a well-earned A.

What it is excellent at: the money engine. A DB-enforced double-entry ledger, a single server-derived idempotent refund path, seam discipline where dev fakes exercise the real path, by-the-book custom auth with a fail-fast secret gate, high-fidelity tests with real browsers + chaos + single-trace coverage and green CI, and documentation rigor most teams never approach. If you want to learn event-driven .NET — outbox, sagas, gateway auth, ledgers — this is a near-ideal reference.

What now remains: external launch gates, not engine work. Live Stripe/Xero and a real ITaxStrategy await a legal entity; an external pen test is pending; prod DB-credential rotation and a secrets store are deferred; one NFR (SSR p95) is proxied rather than measured at the render boundary; and the kind-deploy CI job is slow and cost-gated. Every one of these is bounded, documented, and outside the ledger.

Who it's a good template for: engineers and teams who want a rigorous, well-documented, now-containerized blueprint for distributed e-commerce in modern .NET — with the remaining work being business onboarding and external assurance rather than core construction.
